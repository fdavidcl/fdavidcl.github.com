Truco: Botones de 2 estados usando solo CSS

Un botón de dos estados es como un interruptor, puede estar activado o desactivado. Para crear y manejar un botón de este estilo, normalmente utilizaríamos Javascript y un par de clases CSS para cambiar entre los dos estados. Esta no es una mala solución, pero con un poco de ingenio podemos obtener el mismo efecto utilizando únicamente el CSS, lo que supone que nos ahorramos esas líneas de Javascript que pueden introducir algo de lentitud en el sitio web.

Primero, preguntémonos, ¿a qué elemento HTML podemos recurrir para obtener un estado booleano (verdadero o falso)? A las checkboxes, es decir, esto:

<input type="checkbox" id="micheck1" />
<label for="micheck1">Una checkbox</label>

Para convertir esto en un botón hace falta mucha modificación por CSS. Voy a tratar de hacer algo sencillo para que sea fácil de adaptar a distintos estilos:

#micheck1 {
	display: none;
}
#micheck1+label {
	display: inline-block;
	border: 2px solid black;
	font-size: 14px;
	padding: 3px 6px;
	background: #eaeaea;
	-webkit-user-select: none; /* Evitamos que se seleccione el texto */
	-khtml-user-select: none;
	-moz-user-select: none;
	-ms-user-select: none;
	user-select: none;
}

Ver el código en acción

Ahora viene la magia. Imagino que ya os iréis suponiendo cómo vamos a convertir esto en un botón de doble estado. Vamos a utilizar el pseudoselector :checked de CSS:

#micheck1:checked+label {
	background: #000;
	color: #fff;
}

Ver el código en acción

Finalmente, un toque de CSS avanzado que nos permite cambiar el texto de la etiqueta según el estado.

#micheck1+label:after {
	content: "Desactivado";
}
#micheck1:checked+label:after {
	content: "Activado!";
}
<input type="checkbox" id="micheck1" />
<label for="micheck1"></label>

Ver el código en acción

¡Y listo! Un botón de dos estados simple utilizando solamente código CSS. Además, es muy sencillo comprobar si está activado o desactivado desde Javascript utilizando document.querySelector("#micheck1").checked

Android de verdad: Nexi, AOSP, Google Editions

Hace unas semanas comentaba que las capas de software que los fabricantes instalan en sus dispositivos Android suelen ralentizar innecesariamente el sistema y además, para mi gusto, lo hacen más feo.

Desde luego, la forma más sencilla (y posiblemente más barata) de conseguir un dispositivo con Android puro es comprar un Nexus. Estos están soportados directamente por Google y tienen siempre la última versión estable de Android. El Nexus 4 (el móvil) está fabricado por LG, la Nexus 7 (tablet pequeña) por ASUS y la Nexus 10 (tablet grande) por Samsung, luego son además dispositivos con un muy buen hardware, que hace mejor la experiencia de este Android puro.

Sin embargo, algunos fabricantes se están empezando a dar cuenta de que ofrecer una experiencia limpia de Android puede ser realmente una ventaja, hablamos concretamente de Sony, Samsung y HTC.

Poco después de mi post anterior, en el Google I/O, Google y Samsung anunciaron el lanzamiento del Galaxy S4 Google Edition, una versión del nuevo dispositivo insignia de Samsung, pero con la última versión de Android, limpio, sin añadidos. Lo que ofrece este móvil es una experiencia Nexus (siempre actualizado y libre de skins) junto con la potencia bruta del hardware de Samsung. Es probablemente de los teléfonos más fluidos y rápidos con Android. Además, curiosamente es más barato que el propio Galaxy S4, pero mientras que el original se puede conseguir con “ayuda” de las operadoras, la Google Edition sólo estará disponible para compra desde Google, y resulta bastante más caro que el Nexus 4, con el que no guarda tantas diferencias.

También se han oído rumores sobre una posible Google Edition del HTC One, un teléfono que ya de por sí está teniendo mucho éxito gracias a sus características novedosas y su cuerpo de aluminio.

Actualización: Google acaba de confirmar que se lanzará el HTC One Google Edition el 26 de junio.

Por último, Sony no está lanzando versiones separadas de sus dispositivos con Android, pero sí que está realizando un esfuerzo enorme de colaboración con la comunidad de desarrolladores, añadiendo a los recientes Xperia S, Xperia Z y Xperia Tablet Z al proyecto AOSP (Android Open Source Project), de forma que cualquiera que posea ya uno de estos dispositivos podrá instalar Android limpio en ellos sin pasar por muchos peligros como sucede en otros dispositivos a la hora de instalar ROMs.

En resumen, podemos conseguir una experiencia stock (de Android puro) en los cuatro mejores teléfonos con Android: el Nexus 4, el Galaxy S4, el One y el Xperia Z, y en la Xperia Tablet Z además de en las tablet Nexus. No puedo más que alegrarme por que los fabricantes estén promoviendo el volver a Android original como, al menos, una opción soportada oficialmente, y espero que vaya a más y que no sólo esté reservado para los de más alta gama, sino también para los gama media.

OEMs, estáis arruinando Android. Usuarios, podéis hacer algo al respecto

Android es sin duda el sistema operativo móvil que alberga más posibilidades de personalización, tanto para los usuarios como para los propios fabricantes (u OEMs). Esto supone una gran ventaja, porque cualquiera con un móvil Android puede convertirlo en una experiencia única, gracias a la amplia customización que proporcionan los launchers (sustitutos del escritorio estándar), pantallas de bloqueo, paquetes de iconos y un largo etcétera de aplicaciones y añadidos.

Atrapados en el pasado

El problema llega cuando no son los usuarios los que utilizan y configuran estas aplicaciones para personalizar su teléfono, sino que son los fabricantes los que deciden por ellos. Son más que conocidas las skins que colocan Samsung, HTC, LG y demás sobre Android, generalmente provocando que el sistema se ralentice y la experiencia de usuario sea inconsistente y confusa: mientras que algunos OEMs, como Sony y HTC, proporcionan una capa ligera de personalización que provoca una distinción del Android normal sin romper la estética general, otros como Samsung y LG prefieren modificar completamente la apariencia, arriesgándose a ofrecer una interfaz incoherente, al no seguir las reglas de diseño que utilizan Android y gran parte de las aplicaciones. No solo eso, sino que los launchers que utilizan estos últimos en sus smartphones parecen anticuados y no se han liberado aún de la estética a lo iOS que tuvo éxito en su momento. Android ha avanzado mucho desde entonces, ahora utiliza un diseño más minimalista y sobrio, adaptable a teléfonos y tablets, mientras que algunos fabricantes se han quedado estancados o simplemente prefieren conservar una interfaz conocida que introducir avances.

Otro problema aún más grave, y en parte consecuencia del anterior, es que los fabricantes dan un soporte muy limitado a la mayoría de smartphones, a excepción solamente de los de gama alta. Esto significa que tras unos meses del lanzamiento de un nuevo teléfono, dejan de proporcionar actualizaciones del sistema (porque llevar a cabo toda la personalización y adaptación de la nueva versión al teléfono cuesta trabajo, parte del cual se ahorrarían si utilizasen un Android puro). Por ejemplo, el Nexus S de Samsung (el teléfono oficial de Google lanzado a final de 2010) tiene actualización oficial a Android 4.1.2 (Jelly Bean) mientras que el Galaxy Ace, también de Samsung y lanzado tres meses después (y lleva vendiéndose hasta hoy), se quedó en 2.3.6 Gingerbread.

Regreso al futuro

Lo bueno de la enorme libertad que proporciona Android es que el usuario puede cambiar parte del sistema (launcher, pantalla de bloqueo y demás) de forma muy sencilla. Existen algunos launchers que ofrecen la interfaz original de Android, en ocasiones con algunas mejoras en la gestión de widgets y en animaciones. Entre los más conocidos están Apex Launcher, Nova Launcher, Trebuchet y Holo Launcher. A destacar de los tres primeros es la cantidad de características adicionales como animaciones de transición entre escritorios, cantidad de iconos por pantalla modificable y un dock muy personalizable. De Holo Launcher hay que señalar que es el único que puede ejecutarse en versiones antiguas (2.x.x) de Android y que trata de traer la nueva interfaz característica de Android a teléfonos antiguos que no soportan esas nuevas versiones, luego es un launcher muy rápido y ligero, que ocupa poco espacio en RAM.

Para usuarios avanzados que quieren ir más allá de la personalización, y quieren actualizar su móvil o tablet a la versión más reciente posible de Android, puede que haya desarrolladores independientes que realicen una ROM para el dispositivo. Instalar una ROM, por desgracia, no suele ser un proceso simple: por ejemplo, en algunos dispositivos el cargador de arranque está desbloqueado o se puede desbloquear de forma oficial, pero en otros hay que utilizar otros métodos para burlar el bloqueo.

Tres de las ROMs más conocidas y extendidas son AOSP, CyanogenMod y Paranoid Android. La primera de ellas es un proyecto que pretende proporcionar un Android completo, puramente de código libre, a la mayor variedad de dispositivos posible. Como se trata de un proyecto open source, requiere que los controladores (drivers) para el funcionamiento del dispositivo sean liberados, por lo que no hay una versión para muchos móviles/tablets. Por otro lado, CyanogenMod ofrece un Android muy cercano al propio de los dispositivos Nexus, con algunas mejoras y posibilidades de personalización extra. Por último, Paranoid Android es una ROM que, mantiendo la estética original del Android puro, introduce avances profundos en la interfaz de usuario, como el sistema de botones PIE que sustituye a los botones Atrás, Inicio y Recientes de la barra inferior, y el sistema de notificaciones Halo, que es una novedosa forma de gestionar las notificaciones.

Aprendiendo del presente

Los fabricantes quieren diferenciarse de los demás, eso es algo lógico y evidente, pero la manera en que lo hacen hoy día solo provoca que ralenticen inútilmente sus teléfonos y tablets, y presenten una apariencia que rompe con la estética de Android. Si de verdad quieren ser diferentes, no hay problema, tienen la libertad de tomar el código de Android y crear su propio sistema operativo móvil, realizando cambios profundos en el sistema que procuren mejoras y cambios en la experiencia del usuario, pero manteniendo compatibilidad con aplicaciones. Esto mismo ya se ha hecho antes, Amazon con sus Kindle Fire (en la imagen, en el centro) y Fusion Garage con los Grid4 (a la derecha) y Grid10 tomaron ese código y crearon un S.O. muy modificado basado en Android.

Por otro lado, ofrecer una experiencia pura de Android sería algo novedoso que hasta ahora solo teníamos con los dispositivos Nexus (a la izquierda en la imagen). Las últimas versiones de Android vienen de serie con una cantidad enorme de funcionalidades y son mucho más rápidas que antes, por lo que es una opción que todo fabricante debería considerar. Parece que Motorola, comprada hace no mucho por Google, comenzará a seguir esta regla en sus próximos teléfonos.

Y respecto a los usuarios, si los OEMs siguen utilizando Android y no ofrecen la experiencia que ellos quieren, son libres de modificar a su antojo las aplicaciones básicas del sistema. Es más, sería conveniente que se extendiera el conocimiento sobre estas posibilidades, que el usuario no tuviera que quedarse anclado en Touchwiz porque utilice un teléfono Samsung, o en Sense porque tenga un HTC. Resumiendo, experimentad con vuestros móviles y tablets, observad cuál es la configuración que mejor funciona o que más os gusta, y compartid la experiencia.

Para dar ejemplo, yo comparto la mía: tengo una Samsung Galaxy Tab 2 7.0 (7 pulgadas, competidora de la Nexus 7 y similares) con CyanogenMod 10.1, que se basa en Android 4.2.2 Jelly Bean, algo que Samsung no ofrece para esta tablet. La rapidez y la fluidez del sistema es mucho mayor que con la versión de fábrica, y además estoy libre de la interfaz incoherente que imponía Samsung, que poseía una estética más tradicional en algunas aplicaciones y más moderna en otros componentes del sistema. Por último, la batería dura ahora muchísimo, gracias en parte a que este Android ocupa en memoria RAM mucho menos que la versión original.

Distribuyendo contenido en columnas mediante CSS

Por lo general, para estructurar un contenido en columnas se utilizan tablas HTML (¡horror!) o bien se utiliza la propiedad float de los elementos para que se distribuyan en filas de forma fluida. Pero si lo que queremos es crear una distribución fluida en columnas, podemos utilizar unas nuevas propiedades de CSS3 que lo permiten.

#exterior {
  column-width: 300px;
}

Esta propiedad permite establecer una guía del ancho de columna requerido. Es importante destacar que las columnas no tendrán por qué tomar el valor exacto de column-width, sino que se adaptarán a la página para conseguir un ancho similar al dado. Con column-width se consigue un número de columnas variable, todas con el mismo ancho, que se adaptan según el tamaño del contenedor (el ancho de un div, el tamaño de la ventana del usuario, etc.). Chrome y Safari requieren el prefijo -webkit-, Firefox utiliza -moz- y Opera e Internet Explorer 10 soportan el estándar.

#exterior {
  column-count: 3;
}

De esta forma se distribuirán las columnas adecuadamente para conseguir el número de columnas pedido. El ancho de cada una será variable, adaptado al elemento contenedor y al número de columnas. De nuevo, Chrome, Safari y Firefox utilizan sus prefijos, mientras que IE10 y Opera aceptan el estándar.

#exterior {
  column-gap: 20px;
  column-rule: 1px solid red;
}

Estas dos reglas configuran el estilo de la división de las columnas, column-gap permite modificar el ancho de separación, y column-rule puede crear una línea a lo largo del espacio entre dos columnas.

#interior {
  break-inside: avoid;
  -webkit-column-break-inside: avoid;
  break-before: column;
  -webkit-column-break-before: always;
}

Estas reglas cambian el comportamiento de las columnas, indicando dónde deben partirse y donde deben evitarlo. En este caso, los navegadores basados en Webkit (Chrome, Safari…) utilizan propiedades alternativas, denominadas -webkit-column-break-before y -webkit-column-break-inside (además, -webkit-column-break-before acepta un valor distinto al del estándar). Firefox no soporta ninguna de estas propiedades aún, pero se puede utilizar display: table; para evitar cortes dentro de un elemento.

Si queréis más información sobre estas propiedades, y verlas en acción, visitad las páginas al respecto de Mozilla Developer Network y QuirksMode.
